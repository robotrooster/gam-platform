## Architecture decision recorded

PaymentIntent verification on this rail is metadata-anchored, not
webhook-anchored. Sequence:

1. Frontend calls `POST /api/background/payment-intent` (or the new
   pool variant) ‚Üí backend creates a Stripe PI with metadata
   `{ kind, userId, [matchId] }` and returns the clientSecret.
2. Frontend confirms the PI client-side via Stripe Elements.
3. Frontend calls `/submit` (or `/purchase-report`) passing the
   intentId. Backend retrieves the PI from Stripe, asserts
   `status='succeeded'`, kind matches, userId matches the caller,
   amount matches, then writes the row.

A webhook would also work, but verifying on submit keeps the flow
linear (no race between PI confirmation and the write) and matches
the metadata semantics: a PI for `background_check_intake` has no
purpose outside the /submit path that consumes it.

Idempotency is database-side: `background_checks.applicant_payment_
intent_id` is UNIQUE-indexed (partial, NULL-tolerant); pool unlock
uses a `WHERE report_fee_paid=FALSE` guard on the UPDATE so a replay
returns 409 instead of silently double-flipping.

Dev mode preserves mock IDs when `STRIPE_SECRET_KEY` is unset ‚Äî the
verifier accepts `pi_intake_mock_*` / `pi_pool_mock_*` prefixes in
non-production and rejects them in production.

  - Mock IDs (`pi_intake_mock_*`, `pi_pool_mock_*`) accepted in
    non-production, rejected in production.
  - Real IDs: retrieves the PI from Stripe, asserts
    `status === 'succeeded'`, `metadata.kind` matches, optional
    `metadata.userId` and `metadata.matchId` match, amount within 1¬¢
    of expected.
- `POST /api/background/payment-intent` ‚Äî replaces the mock with a
  real `paymentIntents.create` (card; metadata `kind=
  background_check_intake`, `userId`, `feeUsd`). Mock fallback when
  Stripe absent.
- `POST /api/background/submit` ‚Äî calls `verifyPaymentIntent` before
  the INSERT. Catches `23505` on `background_checks_applicant_pi_uniq`
  and returns 409 with a clear message. Inserts intent_id alongside
  the row.
- `POST /api/background/pool/match/:matchId/payment-intent` ‚Äî new
  endpoint, mirror of the applicant flow. Returns clientSecret +
  intentId for the $1 pool report unlock fee.
- `POST /api/background/pool/match/:matchId/purchase-report` ‚Äî now
  requires `paymentIntentId` in body. Verifies PI (metadata
  `kind=pool_report_unlock`, `userId`, `matchId`, amount), then flips
  `report_fee_paid` with a `WHERE report_fee_paid=FALSE` guard. 409
  on race / replay.

## What this session did NOT do

- **No frontend Stripe Elements wiring.** The applicant intake page +
  the landlord pool match modal both still call the existing API
  shape but neither is wrapped in `<Elements>` to actually confirm
  the intent client-side. They'll fail on real Stripe (no
  `paymentMethod` confirms) until that wires up. Mock dev path keeps
  working since the verifier short-circuits mock IDs.
- **No webhook for these PI types.** `webhooks.ts` only updates
  `payments` / `disbursements` rows. A `background_check_intake` /
  `pool_report_unlock` PI succeeding via webhook is a no-op. That's
  intentional ‚Äî verification happens at submit time. If we want a
  paid-but-never-submitted refund flow, that's a separate session.
- **No audit of remaining pi_* mock IDs elsewhere.** Quick grep found
  none, but a thorough pass across services + jobs is its own task.
- **No OTP enablement infra.** Listed as outstanding under Item 16
  batch 3+ in DEFERRED. Different shape (SetupIntent for ACH, not
  PaymentIntent for one-shot card).
