NEW STANDING RULES

S57-1: zsh interactive_comments is off by default. The shell does NOT honor # as a comment character in interactive paste mode. A line like `# === foo ===` becomes a command attempt where zsh tries to execute `# ===` and prints "== not found". Cost us roughly 30 minutes mid-session. Going forward: paste blocks for terminal use have ZERO `#` comments at the shell level. Comments live inside heredocs (where the body is opaque to zsh) or get dropped. If a comment is genuinely needed, use `: # comment` form which zsh evaluates as the no-op builtin `:` ignoring the rest.

S57-2: psql output goes through the pager unless explicitly bypassed. Default PAGER on macOS is `less`, which paginates anything over a screenful and breaks scripted recon by stopping mid-output at a `:` prompt. Going forward: every psql invocation in a scripted block prefixes `PAGER=cat`. Example: `PAGER=cat psql -d gam -c "\d properties"`.

S57-3: Python r-string triple-quote heredoc is THE file-write pattern for paste blocks. The pattern is `python3 - <<'PYEND'` with file content assigned to a variable using r-prefix and the triple-quote string form. Then write via `open(path,'w').write(var)`. The single-quoted heredoc delimiter prevents shell expansion of the body. The r-prefix prevents Python escape-sequence interpretation. The triple quotes preserve newlines verbatim. This pattern survives content with backticks, dollar signs, em dashes, single quotes, and anything else that would have broken older heredoc-and-cat patterns. Edge case caught while writing this very handoff: the file content itself cannot contain a literal triple-quote sequence — if it must, build that sequence at runtime with a TQ variable and concatenate. Use this pattern for ANY file write longer than 5 lines.

S57-4: Don't trust handoff memo numbers about disk artifacts — recon first. The S56 deferred memo said "3 backup files cleanup in routes/" and reality was 19 files spanning 13 route files across 3 sessions, ~436KB. Same shape as the S29c-2-E claim about a Python diff harness that S56 caught (didn't exist on disk). Pattern: when a memo references something on disk with a number, ls before scoping. Memo numbers are hand-written; disk truth isn't.

S57-5: macOS BSD rm -v output differs from GNU. zsh on macOS uses BSD `rm`, which prints bare paths to stdout when -v is passed. GNU `rm -v` prints `removed 'path'`. If a verification step downstream checks for the GNU format string, it'll fail on macOS. The deletes still happen — verification just needs to read what BSD prints (bare paths) or check via `ls` after. Same general note for `cp -v`, `mv -v`.

S57-6: The harness only scans INSERT and UPDATE. SELECT-only drift (a route that queries a phantom table without writing to it) will NOT show in harness output. Discovered while ripping utility.ts: the route had only a SELECT against utility_bills, harness had nothing to flag. Implication: harness is necessary but not sufficient — it catches the dangerous category (silent write no-ops, $9-style param renumbering bugs) but won't catch every category. Adding SELECT scanning to the harness is itself deferred work; not a priority because SELECT failures are loud at runtime (query errors with shape mismatches) where INSERT/UPDATE failures can be silent (column doesn't exist sometimes throws, sometimes the surrounding ON CONFLICT or COALESCE swallows it).

S57-7: Chat client autolinks dotted strings that look like domains. When recon output renders things like req.params.id or p.id as hyperlinks (with bracketed-URL syntax wrapping them), that's the chat UI rendering, NOT a real URL in the file. Don't chase phantom bugs because of formatting in the recon output. Always trust what's on disk over what the chat shows. The autolinking can also hit filenames in shell commands; if a paste block fails with a no-such-file error pointing at a bracketed-URL filename, the issue is at the paste boundary, not on disk.
