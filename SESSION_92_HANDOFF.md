## Architecture decision recorded

**Product direction: build the master schedule.** Per Nic at S92 —
"absolute must." That moots the finish-or-strip question and locks
in unit_bookings as a launch product. Frontend SchedulePage.tsx +
booking modal + the booking-flow routes I gated in S81 all become
load-bearing surface area.

**Vocabulary winner: route + frontend.** Three layers had drifted:
- schema `unit_bookings.booking_type` CHECK: `nightly | weekly | lease_hold`
- route INSERT: writes `lease_type` (column didn't exist) with values
  `nightly | weekly | month_to_month | long_term`
- frontend `SCHEDULE_BOOKING_TYPES`: `nightly | weekly | month_to_month |
  long_term`

Route + frontend already agreed; schema was the odd one out. S92
collapses to the route+frontend vocabulary (`lease_type`, five values).
`lease_hold` from the old schema CHECK is preserved as a fifth value
for the future "block off this spot for an existing tenant's lease"
semantic — no current consumer, but the value is harmless and avoids
a future migration if that use case lands.

**unit_bookings.landlord_id is required and FK-enforced.** Despite
being derivable from `unit_id → units.landlord_id`, it's stamped on
the row directly because the master-schedule query
(`/units/schedule/master`) filters on it heavily. The new
`(landlord_id, check_in, check_out)` index lights up that query
without requiring a join through units.

**lease_types_allowed default goes from `['fixed_term']` to `{}`.**
The pre-S92 default was a vocabulary that didn't match anywhere else
in the codebase — dead value that would have leaked into UI dropdowns.
New units start empty; the `PATCH /:id/type` route fills it via the
`LEASE_TYPE_MATRIX` keyed on unit_type (units.ts:184-189).

### Migration 20260503130000_master_schedule_finish.sql

**units (3 cols added):**
- `monthly_rate numeric(10,2)` — for monthly-rate display alongside
  nightly/weekly on short-term units.

- `max_stay_nights integer` — landlord-configurable maximum stay.

- `DROP CONSTRAINT unit_bookings_booking_type_check`
- `DROP COLUMN booking_type`
- `ADD landlord_id uuid NOT NULL REFERENCES landlords(id) ON DELETE RESTRICT`
- `ADD lease_type text NOT NULL` with CHECK covering five values
- `ADD nightly_rate numeric(10,2)` — snapshot at booking time
- `ADD weekly_rate numeric(10,2)` — snapshot at booking time
- `ADD platform_fee numeric(10,2) NOT NULL DEFAULT 0` — calculated
  in the route as 5% of total_amount
- `ADD source text NOT NULL DEFAULT 'direct'` — channel attribution
  (direct, future: airbnb, vrbo, etc.)

**Indexes:**
- `idx_unit_bookings_landlord_dates` on `(landlord_id, check_in,
  check_out)` for the schedule/master range query.
- `idx_unit_bookings_unit_dates` partial on `(unit_id, check_in,
  check_out) WHERE status NOT IN ('cancelled', 'no_show')` — used
  by the conflict-detection query in `POST /:id/bookings`.
