## Architecture decision recorded

**One adverse-action notice per background check.** UNIQUE on
`background_check_id`. Re-deciding (denied → re-review → denied
again) doesn't double-insert; the original notice is the legal record
of what was sent on which date. The first send is durable.

**Notice text is stored verbatim, not regenerated.** Template changes
must not retroactively mutate what an applicant actually received. The
`notice_text` column holds the exact rendered string at send time so
any future regulator inquiry or applicant dispute can be answered with
the actual document, not an approximation rebuilt from current code.

**CRA disclosure lives on the provider, not on the landlord.** Each
background-check provider IS a consumer reporting agency for FCRA
purposes. The provider knows its own contact info; the route gets it
via `provider.craDisclosure()`. When a real adapter (Checkr, RentPrep,
SafeRent, etc.) lands, it returns its real CRA contact info and every
notice automatically picks it up. Mock provider returns a clearly-
labeled placeholder so a notice generated against the mock can never be
mistaken for a real one.

**Federal baseline only.** State-specific adverse-action requirements
(California Civil Code §1786.40 has additional fields; some states
require different timing or notice formats) are NOT in this table. Per
the GAM "no state-specific legal logic" rule, those are landlord-
configurable extensions that compose on top of the federal text. When
a landlord operates in a state with stricter requirements, the
configuration carries the additional disclosures.

**Failure-tolerant denial path.** Email send and notice insert run in
a try/catch wrapped around the denial. If either fails (Resend down,
DB conflict, etc.), the denial itself still completes and the failure
is logged for admin reconciliation. Blocking the denial on email
delivery would be worse for the landlord and the applicant.

## What this session did NOT do

- **No legal review.** The notice copy in `lib/adverseAction.ts` is
  pulled from standard FCRA compliance templates. Final wording should
  be reviewed by your housing-law counsel before go-live, especially
  the "Decision summary" framing (landlord-supplied free text) which
  is the highest-risk field.
- **No state-specific notices.** California, New York, and a handful
  of other states have stricter adverse-action regimes. Add via
  landlord-configurable extensions when GAM serves landlords there.
- **No frontend.** Denial UI doesn't yet show the notice was sent.
  Audit endpoint exists; just needs a "View Adverse Action Notice"
  button on the bg check detail page.
- **No real CRA disclosure values.** Mock provider's placeholder
  values are intentionally obviously-fake. Real values land when the
  Checkr (or alt) adapter does.
- **No "pre-adverse action" notice.** Some workflows require sending
  a 5-day-warning before the formal adverse action; that's a separate
  step and not federally required for housing (it's required for
  employment under FCRA §604(b)(3)). Not in scope.
