# Session 187 — closed

## Theme

Quarter-session schema cleanup carried over from S186: align
`onsite_manager_scopes` with the other two scope tables by adding
the `all_properties` boolean column. Removes the S185 empty-array
workaround in `routeMaintenanceNotification` and gives the table
an explicit semantic for blanket coverage.

## What S187 shipped

### Migration `20260508120000_onsite_manager_all_properties.sql`

```sql
ALTER TABLE onsite_manager_scopes
  ADD COLUMN all_properties boolean NOT NULL DEFAULT false;

UPDATE onsite_manager_scopes
   SET all_properties = true
 WHERE property_ids = '{}'
   AND unit_ids = '{}';
```

Backfill rule preserves the pre-S187 semantic that the S185
maintTeam query was honoring: an onsite manager with empty
`property_ids` + empty `unit_ids` was the implicit "all properties
under this landlord" assignment per the S80 design. Post-migration
that meaning is encoded in the column directly. Dev DB had 0 rows
in the table; backfill was a no-op runtime but the SQL is correct
for any production data.

Applied via `npm run db:migrate`. `schema.sql` regenerated.

### Backend wiring

- `routes/scopes.ts:37-40` — `osScopeSchema` adds

  `pmScopeSchema` and `mwScopeSchema`.
- `routes/scopes.ts:101-108` — `insertScopeRow` for `'onsite_manager'`

- `services/notifications.ts` `routeMaintenanceNotification`

  instead of the S185 empty-array convention. Comment cleaned up.

### What was deliberately NOT changed

- **`PATCH /api/scopes/:roleType/:userId` for `onsite_manager`** —
  the UPDATE statement still does NOT touch `all_properties`. If
  the frontend POSTs without `allProperties`, zod defaults it to
  false, which would zero out an existing `true` row on save.
  Preserving the existing column value via "don't write it" is the
  safe minimal-breakage path. Migration backfill is the sole writer
  of `all_properties = true` on existing rows; new rows can be
  created with it via INSERT through the existing invitation flow.
  If Nic later wants `allProperties` to be flippable on existing
  onsite managers via the team UI, that's a one-line UPDATE
  addition + frontend toggle — separate session.

- **TeamPage frontend** — doesn't currently render or edit
  scope-detail fields for onsite managers (only role + permissions).
  Adding `allProperties` to the GET response is forward-compat;
  surfacing it in UI is a separate UX call.

## Decisions made (S187)

| Question | Decision |
|---|---|
| Backfill `all_properties = true` for empty-array rows, or default everyone to false? | Backfill true. Pre-S187 the empty-array convention was honored by the S185 maintTeam query as "all properties under landlord." Switching that to false would silently break maintenance routing for any prod onsite manager who relied on the convention. Dev DB had 0 rows so this was theoretical; correct for any prod data. |
| Surface `allProperties` in PATCH `/scopes/:roleType/:userId`? | No. Without a frontend send-allProperties path, the zod default-false would reset `true` rows on every save. Keep PATCH out of `all_properties` until the frontend is ready. |
| Add a toggle UI on TeamPage for onsite-manager allProperties? | No, deferred. TeamPage doesn't currently render onsite scope details; adding the toggle is a separate UX call. The data model is consistent now; UI catches up when needed. |
| Update the S185 PERMISSIONS_AUDIT.md note about the empty-array workaround? | No, leave it. The audit captures the historical state ("S185 worked around the missing column"); marking it superseded would muddy the audit trail. The notifications.ts code change is the canonical record that the workaround is gone. |
