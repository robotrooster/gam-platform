## Architecture decision recorded

**Card sales record-back is a separate POST, not a webhook.** The
terminal capture flow is now five steps: connection-token →
create-payment-intent → reader collects → capture → record. The
final record-back hits the same `POST /api/pos/transactions` that
cash and charge sales already use; the only difference is the
`paymentMethod: 'card'` body field and the new
`stripePaymentIntentId` field. Webhook-driven record-back was
considered and rejected — the synchronous capture-then-record flow
keeps the cashier UI's success state authoritative (no race where
the receipt prints before the row exists). Same posture as the S83
applicant-bg-check verification on submit.

**Idempotency at the DB.** Partial UNIQUE on
`pos_transactions.stripe_payment_intent_id WHERE NOT NULL` —
identical to the pattern in background_checks (S83) and
pool_match_requests (S83) and utility_bills (S90). The route catches
23505, fetches the existing row, returns 200 with a
"Transaction already recorded" message rather than a hard 409
because retries are an expected flow (network hiccup between capture
and record-back). Same row, no double-charge.

**Smoke walk confirms schema-route alignment.** S93 added 13 tables
matching pos.ts column-by-column; S94 ran the actual INSERT shapes
(verbatim from each route handler) inside a rolled-back transaction.
All 14 inserts succeeded — vendor, category, item, price_history,
variant, transaction, transaction_items, inventory_log, tax_rate,
discount, refund, PO, PO item, plus the second card transaction with
PI id. PI duplicate test fired the expected 23505 on
pos_transactions_stripe_pi_uniq. Final stock_qty went from 10 → 9
correctly.

## What this session did NOT do

- **No frontend.** POS app + Stripe Terminal SDK integration on the
  cashier device is its own multi-day product surface. The backend
  is now ready to receive `paymentMethod: 'card'` + PI id from
  whatever frontend writes the cashier UI.
- **No webhook on `payment_intent.succeeded` for card-present.**
  Considered and rejected (see architecture note above) — synchronous
  record-back is cleaner. If we ever need webhook-side reconciliation
  (e.g., for a separate "settled" status flip), the
  stripe_payment_intent_id column is the join key.
- **No Stripe Terminal seed data.** Smoke walk used `paymentMethod:
  'cash'` for the main chain and a synthetic `pi_smoke_test_123` for
  the dup test. Real Terminal SDK exercise needs hardware.
- **No end-of-day reconciliation cron.** Still on the Item 14
  "outstanding" list. Logical follow-up once the cashier UI exists
  and there's data to roll up.
