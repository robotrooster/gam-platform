## Architecture decision recorded

**Late-detection emails fire every cycle, fail-soft.** Each overdue
payment triggers a `sendLatePaymentNotice` to the landlord and (on
the 2nd late payment) a one-shot `sendOnTimePayInvitation` to the
tenant. Email failures log + continue — one bad address must not
abort the rest of the loop. The SELECT was widened to fetch tenant
email/name + landlord business name + property name in the same join
so each row is self-sufficient (no per-row sub-fetches inside the
loop).

**Work-trade schema matches the existing route shape exactly.** The
routes were written against assumed columns; this migration makes
those assumptions real with the right CHECK constraints, FK directions,
and the UNIQUE that the route's
`INSERT ... ON CONFLICT DO NOTHING` already relied on. No backfill —
the table is empty. Routes are already gated (S81 perm pass).

**Periods are landlord-reconciled, not auto-rolled.** The route
`POST /:id/reconcile` opens the next period as part of closing the
current one. No cron rolls periods on the 1st — that was a
deliberate choice in the route logic and the schema accommodates it
(periods get inserted on demand, not pre-populated).

## Shipped

### apps/api/src/jobs/scheduler.ts
- Imports `sendLatePaymentNotice` + `sendOnTimePayInvitation` from
  services/email.ts (the templates ported in S85).
- Late-payment SELECT widened: now joins `users tu` (tenant), `properties pr`,
  and pulls `tu.email`, `tu.first_name`, `tu.last_name`,
  `landlord business_name COALESCE first/last`, and `pr.name`.
- Per-overdue-row loop calls `sendLatePaymentNotice` with computed
  `daysLate`, then conditionally calls `sendOnTimePayInvitation` when
  the OTP-invite sentinel flips on. Both wrapped in try/catch — email
  failures log + continue.

### Migration 20260503090000_work_trade_subsystem.sql
- `work_trade_agreements` — anchor table. trade_type CHECK
  (full/partial/credit), status CHECK (active/paused/ended), ytd_value
  + flag_1099 + tax_year for the $600 1099 threshold tracking already
  consumed by workTrade.ts:222-227 + reports.ts work-trade-1099
  endpoint.
- `work_trade_logs` — per-shift submissions. status CHECK
  (pending/approved/rejected). `reviewed_by` FK to users with
  ON DELETE SET NULL (preserves the log if the reviewer's user row is
  ever removed).
- `work_trade_periods` — monthly commitment + actuals. UNIQUE
  (agreement_id, period_month, period_year) makes the route's
  ON CONFLICT DO NOTHING actually conflict against something.
  period_month BETWEEN 1 AND 12 CHECK.
- Indexes: agreements by (landlord_id, created_at DESC) +
  tenant_id + unit_id; logs by (agreement_id, work_date DESC) +
  partial on pending; periods by (agreement_id, year DESC, month DESC).
